# 2026-04-03 Windows 桌面 App 設計

## 目標

將目前的台股模擬交易系統做成 `Windows 專用、完全自包含` 的桌面 App，使用者安裝後不需要另外安裝 Python，也不需要手動啟動前後端。

桌面版的核心體驗如下：

- 雙擊 App 後，只開啟前端主視窗
- Python 後端在背景自動啟動
- 關閉視窗時，前後端一起完全結束
- 若背景後端啟動失敗，主畫面仍開啟，並顯示連線異常與重試入口
- 第一版沿用目前專案目錄與 `.env` 邏輯，不搬移設定結構
- 發佈形式為 `Windows 一般安裝版`，有開始選單捷徑與可解除安裝

## 採用方案

採用 `Tauri + Python sidecar`。

理由：

- 前端已經是 `React + Vite`，可直接沿用現有構建流程
- Tauri 較 Electron 更輕量，較適合長時間開啟的看盤型介面
- Tauri 支援 sidecar binary，適合將 Python 後端一起打包
- 能做出標準 Windows 安裝包，符合目前需求

不採用：

- `Electron`：可行但包體較大、資源占用較高
- `PySide / Qt`：需要重做前端，成本過高

## 桌面版架構

桌面版分成 3 層：

1. `Tauri Shell`
   - 建立 Windows 視窗
   - 載入前端資產
   - 管理 Python sidecar 的啟停
   - 提供「重試後端」與「健康狀態輪詢」能力

2. `React Frontend`
   - 延用現有 Vite build 輸出
   - 啟動後先顯示主畫面
   - 若後端未就緒，保留 UI，但以連線異常狀態顯示
   - 提供重試按鈕，觸發 Tauri 重新啟動 sidecar

3. `Python Backend Sidecar`
   - 封裝目前 `run.py` 為桌面版後端入口
   - 維持目前 WebSocket / collector / bridge 模式
   - 由桌面殼層在背景啟動與關閉

## 啟動流程

### 正常啟動

1. 使用者開啟桌面 App
2. Tauri 視窗先顯示前端
3. Tauri 在背景啟動 Python sidecar
4. Python sidecar 讀取目前專案目錄下的 `.env`
5. Python sidecar 啟動 WebSocket / bridge / mock 或實盤元件
6. 前端透過既有連線邏輯連上本地後端
7. UI 進入正常狀態

### 啟動失敗

1. 前端主畫面仍然顯示
2. 連線狀態顯示為異常或待命
3. UI 顯示「重試連線 / 重啟後端」操作
4. 點擊後由 Tauri 再次嘗試啟動 Python sidecar

### 結束流程

1. 使用者關閉主視窗
2. Tauri 發送停止訊號給 Python sidecar
3. 若 sidecar 未在時限內結束，Tauri 強制結束
4. 不保留背景程序，不縮到系統匣

## Python Sidecar 策略

第一版不直接把整個專案當成可編輯的 Python 原始碼環境交給使用者，而是做成：

- 一個桌面版專用 Python 啟動入口
- 由打包流程攜帶 Python runtime
- 啟動時執行既有後端流程

建議做法：

- 新增桌面版啟動檔，例如 `desktop_backend.py`
- 由它負責：
  - 載入 `.env`
  - 設定工作目錄
  - 啟動 `run.py` 對應流程
  - 處理桌面環境下的 stdout/stderr 與退出碼

這樣可避免直接把 Tauri 綁死在目前開發腳本細節上。

## `.env` 與工作目錄策略

第一版沿用目前專案目錄與 `.env`。

具體規則：

- 桌面 App 啟動 sidecar 時，工作目錄指向 App 內部打包後的後端工作區
- sidecar 先找同層 `.env`
- 若找不到，再找使用者可覆蓋位置
- 第一版不做設定頁改寫 `.env`

這樣可以最快落地，但要在桌面版文件中明確說明 `.env` 放置方式。

## 前端調整需求

桌面版前端需要補 3 個能力：

1. `後端狀態顯示`
   - 區分：啟動中、已連線、重連中、啟動失敗

2. `重試入口`
   - 提供使用者按鈕觸發 Tauri 重新啟動後端

3. `桌面環境偵測`
   - 若在 Tauri 中執行，優先走桌面殼層提供的控制流程
   - 若在瀏覽器開發模式，仍保留原本 Web 開發流程

## 安裝包輸出

桌面版輸出為 Windows 一般安裝版：

- 有安裝精靈
- 有開始選單捷徑
- 可從 Windows 移除程式中解除安裝

第一版不要求：

- 自動更新
- 系統匣常駐
- 多實例管理
- 跨平台打包

## 風險與對策

### 1. Python sidecar 打包複雜

風險：
- Python runtime、套件、`.env`、資源檔需要一起帶入

對策：
- 先做最小可行桌面版
- 明確指定後端入口與依賴清單
- 在打包前先固定 Python 啟動方式

### 2. 後端啟動慢導致 UI 看起來像壞掉

風險：
- 使用者誤以為 App 當掉

對策：
- 前端永遠先開
- 顯示「背景啟動中」狀態
- 提供重試按鈕

### 3. `.env` 與執行路徑不一致

風險：
- 桌面版抓不到 API key 或設定

對策：
- 封裝單一桌面版後端入口
- 明確定義 `.env` 搜尋順序
- 補啟動期錯誤訊息

## 驗收標準

桌面版第一版完成時，至少要滿足：

1. 可產出 Windows 安裝包
2. 安裝後可從開始選單啟動
3. 啟動後只看到前端主視窗，沒有額外 console 視窗
4. 背景 Python 後端可自動啟動
5. 前端可正常連上本地後端
6. 關閉主視窗後，前後端程序都會結束
7. 後端啟動失敗時，前端仍可開啟並顯示重試入口

## 實作拆分

建議分 3 段做：

### 第 1 段：桌面殼層建立

- 初始化 Tauri
- 成功載入現有前端 build
- 能在 Windows 開啟主視窗

### 第 2 段：後端 sidecar 整合

- 建立桌面版 Python 啟動入口
- 由 Tauri 背景啟動 sidecar
- 關閉 App 時一起終止

### 第 3 段：前端桌面狀態整合

- 顯示後端狀態
- 支援重試後端
- 補齊桌面模式下的錯誤提示
